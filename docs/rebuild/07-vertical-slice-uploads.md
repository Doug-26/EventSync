# Phase 07 — Vertical Slice 3: Image Uploads

**Goal:** add cover-image upload to events. Backend exposes `POST /api/v1/uploads/images` with magic-byte validation; frontend wires it into the create/edit forms with preview and drag-and-drop. By the end you can upload a JPEG/PNG/WebP cover photo and see it on the event detail page.

## Why this is its own slice

The Events slice (06a/06b) already had a `coverImageUrl` field. What it *didn't* have was a way to put an image there. Splitting upload into its own slice keeps the concerns separate:

- **Events slice** owns the business object (a row in `Events`).
- **Uploads slice** owns the file lifecycle (bytes on disk, validation, URL generation).

Three security concerns shape every decision in this phase:

1. **Content-Type spoofing.** A browser will happily send `Content-Type: image/png` for a PHP shell renamed to `shell.png`. Trusting the header alone is how servers get pwned.
2. **Stored XSS via SVG.** SVG is XML; XML can contain `<script>`. We **never** accept SVG.
3. **Directory traversal.** The client-supplied filename can contain `../../../etc/passwd`. We **never** use it.

The mitigations: **magic-byte signature check**, **MIME whitelist**, **GUID filenames**.

## Step 1: Plan the on-disk layout

```
server/EventSync.Api/
├── wwwroot/
│   └── uploads/
│       └── events/         # one file per upload, GUID name
└── Features/Uploads/
    └── UploadEndpoints.cs  # single endpoint + magic-byte logic
```

URLs returned by the API look like `/uploads/events/{guid}.jpg` — relative paths. The browser resolves them against the API origin (`http://localhost:5000`) because the static-file middleware (added in step 3) maps that URL prefix to the physical folder.

> **Why under `wwwroot/`?** ASP.NET Core's `UseStaticFiles()` defaults to that folder. Putting uploads outside `wwwroot/` is more secure if uploads should not be web-accessible, but here we *want* them served as plain files, so `wwwroot/` is correct.

## Step 2: Add `UploadEndpoints.cs`

Create `server/EventSync.Api/Features/Uploads/UploadEndpoints.cs`:

```csharp
// Generated by GitHub Copilot
namespace EventSync.Api.Features.Uploads;

/// <summary>
/// Maps the image-upload minimal-API endpoint onto the application route table.
/// Uploaded files are persisted to <c>wwwroot/uploads/events/</c> and served as
/// static files. Only authenticated users may upload.
///
/// Security:
/// - 5 MB hard limit via <c>RequestSizeLimit</c>.
/// - Whitelist of raster MIME types (<c>image/jpeg</c>, <c>image/png</c>, <c>image/webp</c>).
/// - Magic-byte validation prevents Content-Type spoofing.
/// - GUID-based file names — user-supplied filename is never used on disk.
/// - Files served with <c>X-Content-Type-Options: nosniff</c> (default Kestrel behaviour).
/// - SVG is explicitly excluded to prevent stored-XSS.
/// </summary>
public static class UploadEndpoints
{
    private const long MaxFileSize = 5 * 1024 * 1024; // 5 MB

    private static readonly Dictionary<string, byte[][]> AllowedSignatures = new()
    {
        ["image/jpeg"] = [new byte[] { 0xFF, 0xD8, 0xFF }],
        ["image/png"] = [new byte[] { 0x89, 0x50, 0x4E, 0x47 }],
        ["image/webp"] = [new byte[] { 0x52, 0x49, 0x46, 0x46 }], // "RIFF"
    };

    private static readonly Dictionary<string, string> MimeToExtension = new()
    {
        ["image/jpeg"] = ".jpg",
        ["image/png"] = ".png",
        ["image/webp"] = ".webp",
    };

    public static IEndpointRouteBuilder MapUploadEndpoints(this IEndpointRouteBuilder app)
    {
        var group = app.MapGroup("/api/v1/uploads")
            .WithTags("Uploads")
            .RequireAuthorization();

        group.MapPost("/images", async (IFormFile file, IWebHostEnvironment env) =>
        {
            // --- Size check ---
            if (file.Length == 0)
            {
                return Results.BadRequest(new { error = "File is empty." });
            }

            if (file.Length > MaxFileSize)
            {
                return Results.BadRequest(new { error = $"File exceeds the {MaxFileSize / (1024 * 1024)} MB limit." });
            }

            // --- MIME type whitelist ---
            var contentType = file.ContentType.ToLowerInvariant();
            if (!AllowedSignatures.ContainsKey(contentType))
            {
                return Results.BadRequest(new { error = "Only JPEG, PNG, and WebP images are allowed." });
            }

            // --- Magic-byte verification ---
            using var stream = file.OpenReadStream();
            var headerBytes = new byte[12]; // longest signature we check is 4 bytes, but read 12 for RIFF+WEBP
            var bytesRead = await stream.ReadAsync(headerBytes.AsMemory(0, headerBytes.Length));
            if (bytesRead < 3)
            {
                return Results.BadRequest(new { error = "File is too small to be a valid image." });
            }

            var signaturesForType = AllowedSignatures[contentType];
            var magicMatch = false;
            foreach (var sig in signaturesForType)
            {
                if (bytesRead >= sig.Length && headerBytes.AsSpan(0, sig.Length).SequenceEqual(sig))
                {
                    // Extra check for WebP: bytes 8-11 must be "WEBP"
                    if (contentType == "image/webp")
                    {
                        if (bytesRead >= 12 &&
                            headerBytes[8] == 0x57 && // W
                            headerBytes[9] == 0x45 && // E
                            headerBytes[10] == 0x42 && // B
                            headerBytes[11] == 0x50)   // P
                        {
                            magicMatch = true;
                        }
                    }
                    else
                    {
                        magicMatch = true;
                    }
                }
            }

            if (!magicMatch)
            {
                return Results.BadRequest(new { error = "File content does not match its declared type." });
            }

            // --- Save to disk ---
            var extension = MimeToExtension[contentType];
            var fileName = $"{Guid.NewGuid()}{extension}";
            var uploadsDir = Path.Combine(env.WebRootPath ?? Path.Combine(env.ContentRootPath, "wwwroot"), "uploads", "events");
            Directory.CreateDirectory(uploadsDir);

            var filePath = Path.Combine(uploadsDir, fileName);
            stream.Position = 0;
            using (var fileStream = new FileStream(filePath, FileMode.Create, FileAccess.Write, FileShare.None))
            {
                await stream.CopyToAsync(fileStream);
            }

            var url = $"/uploads/events/{fileName}";
            return Results.Ok(new { url });
        })
        .WithName("UploadImage")
        .WithSummary("Upload a cover image (JPEG, PNG, or WebP, max 5 MB). Returns the relative URL.")
        .DisableAntiforgery()
        .Accepts<IFormFile>("multipart/form-data")
        .Produces(StatusCodes.Status200OK)
        .Produces(StatusCodes.Status400BadRequest)
        .Produces(StatusCodes.Status401Unauthorized);

        return app;
    }
}
```

### Why each part exists

**The signature dictionary (`AllowedSignatures`)** — every common image format starts with a fixed byte pattern. These are the "magic numbers" you can verify on disk with a hex editor:

| Format | Bytes (hex)           | Meaning |
|--------|----------------------|---------|
| JPEG   | `FF D8 FF`           | SOI marker + first byte of next segment |
| PNG    | `89 50 4E 47`        | `\x89PNG` (the `\x89` is intentionally non-ASCII to detect text-mode corruption) |
| WebP   | `52 49 46 46` then `57 45 42 50` at bytes 8-11 | `RIFF....WEBP` (it's a RIFF container) |

**The 12-byte buffer** — we always read 12 bytes even though JPEG only needs 3, because WebP's identifier is *inside* the RIFF container at offset 8–11. Reading 12 bytes once is simpler than a length-per-format read.

**The WebP nested check** — RIFF (`52 49 46 46`) is also used by AVI, WAV, ANI, and others. Without the `WEBP` check at offset 8, an attacker could upload a malicious `.avi` claiming `image/webp` and we'd write it to disk with a `.webp` extension. The inner check pins it down.

**`Guid.NewGuid()` for the filename** — the *client's* filename is never used. This blocks:
- Directory traversal (`../../etc/passwd`)
- Reserved Windows names (`CON.png`, `NUL.jpg`)
- Collisions (two users uploading `cover.png`)
- Filename-based XSS (`<script>.png` in a directory listing)

**`stream.Position = 0`** — the 12-byte read consumed those bytes. Resetting the position lets `CopyToAsync` write the full file (including the header we just verified).

**`DisableAntiforgery()`** — minimal-API multipart uploads otherwise demand the antiforgery cookie. For this endpoint we rely on the JWT bearer token for auth + the magic-byte check for content safety. (In a session-cookie app you'd keep antiforgery on.)

**`.Accepts<IFormFile>("multipart/form-data")`** — tells OpenAPI/Swashbuckle to render a file-picker in Swagger UI.

> **Why no in-memory image decoding?** A safer-still approach is to fully decode the image (using ImageSharp/SkiaSharp) and re-encode it. That blocks even malformed-but-valid-prefix files. We skip it here to keep the dependency footprint small and because the threat model (an authenticated user uploading their own cover photo) is low. Mention it in interviews as a "next step" if asked.

## Step 3: Wire static files + the endpoint in `Program.cs`

Two changes in `Program.cs`. First the using:

```csharp
using EventSync.Api.Features.Uploads;
using Microsoft.Extensions.FileProviders;   // PhysicalFileProvider
```

Then in the middleware section, **right after `UseSwagger`** (still before `UseRouting`/`UseAuthentication`):

```csharp
// Serve uploaded images from wwwroot/ (e.g. /uploads/events/abc.jpg).
app.UseStaticFiles();

// Explicitly map "/uploads" to the physical wwwroot/uploads folder. This
// ensures the uploads path is served even if the default webroot mapping
// behaves differently in some hosting environments.
var uploadsPhysicalPath = Path.Combine(app.Environment.ContentRootPath, "wwwroot", "uploads");
if (!Directory.Exists(uploadsPhysicalPath))
{
    Directory.CreateDirectory(uploadsPhysicalPath);
}

app.UseStaticFiles(new StaticFileOptions
{
    RequestPath = "/uploads",
    FileProvider = new PhysicalFileProvider(uploadsPhysicalPath)
});
```

And in the endpoint section (with the other `Map…Endpoints()` calls):

```csharp
app.MapUploadEndpoints();
```

### Why two `UseStaticFiles()` calls?

- The first (parameterless) call serves anything under `wwwroot/` at the matching URL. Standard convention.
- The second pins `/uploads` explicitly to `wwwroot/uploads/`. Belt-and-braces: some hosting setups (containers with read-only roots, reverse proxies that rewrite paths) misbehave with the default mapping. The explicit `RequestPath + FileProvider` removes ambiguity.

The `Directory.CreateDirectory(uploadsPhysicalPath)` call ensures the folder exists at startup so the first upload doesn't 404 trying to write to a non-existent path.

uploadImage(file: File): Observable<{ url: string }> {
  const formData = new FormData();
  formData.append('file', file);
  return this.runWithState(
    this.http.post<{ url: string }>(`${this.uploadsUrl}/images`, formData),
  );
}

And `uploadsUrl` is defined as:

```typescript
private readonly uploadsUrl = `${environment.apiUrl}/uploads`;
```

> **Pitfall — manual Content-Type:** **never** set `Content-Type: multipart/form-data` yourself. The browser generates the boundary string and appends it (`multipart/form-data; boundary=----WebKitFormBoundary...`). If you set it manually you lose the boundary and the server can't parse the body.

## Step 6: Wire upload into `create-event.component.ts`

Add these three protected methods + one private helper to the create form. Place them right above the existing `removeCover()` method:

```typescript
/** Handles cover image file selection. Uploads immediately and stores the returned URL. */
protected onFileSelected(event: Event): void {
  const input = event.target as HTMLInputElement;
  const file = input.files?.[0];
  if (!file) return;
  this.uploadFile(file);
}

/** Handles drag-and-drop onto the cover image zone. */
protected onDrop(event: DragEvent): void {
  const file = event.dataTransfer?.files?.[0];
  if (!file) return;
  this.uploadFile(file);
}

private uploadFile(file: File): void {
  const allowed = ['image/jpeg', 'image/png', 'image/webp'];
  if (!allowed.includes(file.type)) {
    this.uploadError.set('Only JPEG, PNG, and WebP images are allowed.');
    return;
  }
  if (file.size > 5 * 1024 * 1024) {
    this.uploadError.set('Image must be under 5 MB.');
    return;
  }
  // Show local preview immediately.
  this.coverPreview.set(URL.createObjectURL(file));
  this.uploading.set(true);
  this.uploadError.set(null);

  this.eventService.uploadImage(file).subscribe({
    next: (res) => {
      this.coverImageUrl.set(res.url);
      this.uploading.set(false);
    },
    error: () => {
      this.uploading.set(false);
      this.uploadError.set(this.eventService.error() ?? 'Upload failed.');
      this.coverPreview.set(null);
    },
  });
}
```

### Why each piece

**Client-side MIME + size check first** — duplicates the backend check, but gives instant feedback (no round-trip). The backend is still the source of truth; the client check is a UX improvement, not a security boundary.

**`URL.createObjectURL(file)`** — creates a synthetic `blob:` URL the `<img>` tag can render *immediately*, before the server returns the real URL. Two signals back the UI:
- `coverPreview` — the blob URL shown while uploading + after success.
- `coverImageUrl` — the *server-returned* URL, submitted with the form.

The form payload only sends `coverImageUrl`. If the upload fails we reset `coverPreview` to null so the broken thumbnail goes away.

**`uploading` signal** — drives a spinner in the template (`@if (uploading()) { <spinner /> }`).

**`URL.revokeObjectURL`** — for a longer-running form you'd want to revoke the blob URL when the file changes to free memory. For a short-lived create form this is omitted.

### Template snippet that drives all of this

Inside `create-event.component.html`, the upload section:

```html
<div
  class="rounded-lg border-2 border-dashed border-slate-300 p-6 text-center"
  (dragover)="$event.preventDefault()"
  (drop)="onDrop($event); $event.preventDefault()"
>
  @if (coverPreview()) {
    <img [src]="coverPreview()" alt="" class="mx-auto max-h-48 rounded" />
    <button type="button" class="mt-2 text-sm text-red-600" (click)="removeCover()">Remove</button>
  } @else {
    <p class="text-slate-600">Drag a JPEG/PNG/WebP here, or</p>
    <input type="file" accept="image/jpeg,image/png,image/webp" (change)="onFileSelected($event)" />
  }
  @if (uploading()) { <app-loading-spinner label="Uploading…" /> }
  @if (uploadError()) { <p class="mt-2 text-sm text-red-600">{{ uploadError() }}</p> }
</div>
```

## Step 8: Display cover image on detail page

The detail page already has a `coverImageSrc` computed signal from phase 06b step 8. It resolves relative URLs against the API origin:

```typescript
protected readonly coverImageSrc = computed(() => {
  const url = this.event()?.coverImageUrl;
  if (!url) return null;
  if (url.startsWith('http://') || url.startsWith('https://')) return url;
  const apiOrigin = environment.apiUrl.replace(/\/api\/v1$/, '');
  return `${apiOrigin}${url}`;
});
```

Template:

```html
@if (coverImageSrc()) {
  <img [src]="coverImageSrc()" alt="" class="aspect-video w-full rounded object-cover" />
}
```

**Why `apiUrl.replace(/\/api\/v1$/, '')`?**

`environment.apiUrl` is `http://localhost:5000/api/v1` so HTTP calls work. The uploaded image URL is `/uploads/events/abc.jpg` — relative to the API **origin** (`http://localhost:5000`), not under the `/api/v1` prefix. The regex strips that suffix so concatenation gives the right URL.

## Step 9: Verify CSP allows images

The `SecurityHeadersMiddleware` (phase 02 step 6) emits a Content Security Policy. Check the `img-src` directive includes `data:` and same-origin loading. The default policy from phase 02 was:

```
default-src 'self'; img-src 'self' data: blob: https:; ...
```

- `'self'` — `/uploads/events/*` served by the same origin as the SPA's API calls.
- `data:` and `blob:` — the local `URL.createObjectURL()` preview.
- `https:` — future support for off-site images (e.g., Auth0 avatars).

If your CSP differs, update the middleware accordingly. Test with the browser DevTools console — any CSP violation logs a `Refused to load the image…` error there.

## Step 10: Production safety review

Before deploying, the upload endpoint has these production concerns. They're called out here so phase 10 (deployment) can address them.

| Concern | MVP status | Hardened production option |
|---|---|---|
| Local disk fills up | No quota | Per-user storage cap + cleanup job for orphaned files |
| Single-host storage | Files live on the API host | Move to S3 / Azure Blob; return CDN URL |
| No virus scan | Files served directly | ClamAV scan before move to web root |
| No image optimization | Original size served to all clients | Resize/recompress with ImageSharp; serve `srcset` |
| Orphan files | Updating an event with a new cover leaves the old file on disk | Background job to delete files not referenced in `Events.CoverImageUrl` |

For interview purposes: be ready to explain that this MVP optimises for **time-to-working-feature** and lists those items as the next steps before going live.
