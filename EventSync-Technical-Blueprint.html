1.4 Key Design Decisions & Tradeoffs

DecisionChoseOverRationale
ArchitectureVertical SliceClean ArchitectureSolo dev + AI: less boilerplate, features are self-contained, easier to generate
State MgmtSignals + ServicesNgRxApp state is simple — no complex cross-feature state sharing needed
AuthAuth0ASP.NET IdentityFree tier, social logins OOTB, less code to maintain, handles OAuth complexity
UITailwind CSSAngular MaterialFull design control, smaller bundle, no opinionated component library to fight
MappingManual / MapsterAutoMapperSimpler debugging, compile-time safety, less magic
API StyleMinimal APIsControllersLess ceremony for Vertical Slice pattern
Token FormatBase64Url(32 random bytes)GUIDGUIDs are guessable (sequential); cryptographic random is not
Why Vertical Slice? Each feature (e.g., "Create Event") is a self-contained unit with its own endpoint, handler, validator, and DTO. This makes it extremely AI-friendly — you can generate an entire feature by describing a single use case, without needing to touch 5 different horizontal layer files.

3.6 Unique Link Generation Logic

  1. On POST /events/{id}/invite-links, generate token: RandomNumberGenerator.GetBytes(32) → Base64Url encode → 43 chars
  2. Store in InviteLinks table with optional expiry and max uses
  3. Return full URL: {frontendBaseUrl}/rsvp/{token}
  4. On GET /invite/{token}, validate: exists, IsActive, not expired, use count within limit
  5. On POST /invite/{token}/rsvp, increment UseCount

4.3 State Management Approach

Angular Signals + Services — no NgRx (overkill for this app's complexity):

  • Each feature service holds reactive state using signal() and computed()
  • EventService exposes signals: events, loading, error
  • toSignal() from @angular/core/rxjs-interop bridges HTTP Observables into signals
  • For real-time RSVP updates, SignalR messages update signal values directly

4.4 Form Validation Strategy

  • Reactive Forms with typed FormGroup (Angular typed forms)
  • Custom validators for: future date, URL format, max attendees ≥ 1
  • Error messages displayed inline via shared ValidationMessageComponent
  • Accessibility: aria-describedby for error messages, proper <label> associations

4.5 HTTP Interceptors

  • authInterceptor (functional): injects Auth0 access token on requests to the API base URL
  • errorInterceptor: catches 401 (redirect to login), 403 (forbidden), 5xx (toast notification)

6. Security Plan

6.1 Input Validation & Sanitization

  • Server-side: FluentValidation on every MediatR request — validates lengths, formats, ranges, required fields
  • Client-side: Angular Reactive Forms validators (for UX only — never trust client)
  • HTML sanitization: User-generated text sanitized on output using Angular's built-in sanitizer; stored as-is for fidelity
  • SQL Injection: Not applicable — EF Core parameterizes all queries

6.2 Rate Limiting on RSVP Endpoint

  • ASP.NET Core built-in RateLimiter middleware
  • Policy: Fixed window, 5 RSVP submissions per IP per 10-minute window on POST /invite/{token}/rsvp
  • Returns 429 Too Many Requests with Retry-After header
  • Secondary protection: duplicate detection by GuestEmail + EventId (unique filtered index)

6.3 Expiring Invite Links

  • ExpiresAt checked on every GET /invite/{token} and POST /invite/{token}/rsvp
  • Expired links return 410 Gone with a friendly message
  • MaxUses compared against UseCount — returns 410 Gone when exceeded
  • Organizer can deactivate link by setting IsActive = false

6.4 CORS Policy

  • Development: Allow http://localhost:4200
  • Production: Allow only the configured frontend domain
  • Restrict methods: GET, POST, PUT, PATCH, DELETE
  • Restrict headers: Authorization, Content-Type
  • No wildcard * origins

6.5 Authentication & Authorization

  • Auth0 JWT validation: .AddAuthentication().AddJwtBearer() with Auth0 domain/audience
  • Authorization policies: [Authorize] on all endpoints except public invite/RSVP routes
  • Resource-based auth: Events can only be edited/deleted by their OrganizerId — checked in handler, not just by policy
  • No admin role for MVP: Organizer owns their events, no cross-user access

  • RSVP submissions logged with IP for abuse investigation (no unnecessary PII retention)

- Prefer type inference when the type is obvious
- Avoid the `any` type; use `unknown` when type is uncertain

## Angular Best Practices
- Always use standalone components over NgModules
- Must NOT set `standalone: true` inside Angular decorators. It's the default in Angular v20+.
- Use signals for state management
- Implement lazy loading for feature routes
- Do NOT use the `@HostBinding` and `@HostListener` decorators. Put host bindings
  inside the `host` object of the `@Component` or `@Directive` decorator instead
- Use `NgOptimizedImage` for all static images.
  - `NgOptimizedImage` does not work for inline base64 images.

## Accessibility Requirements
- It MUST pass all AXE checks.
- It MUST follow all WCAG AA minimums, including focus management, color contrast,
  and ARIA attributes.

## Components
- Keep components small and focused on a single responsibility
- Use `input()` and `output()` functions instead of decorators
- Use `computed()` for derived state
- Set `changeDetection: ChangeDetectionStrategy.OnPush` in `@Component` decorator
- Prefer Reactive forms instead of Template-driven ones
- Do NOT use `ngClass`, use `class` bindings instead
- Do NOT use `ngStyle`, use `style` bindings instead
- When using external templates/styles, use paths relative to the component TS file.

## State Management
- Use signals for local component state
- Use `computed()` for derived state
- Keep state transformations pure and predictable
- Do NOT use `mutate` on signals, use `update` or `set` instead

## Templates
- Keep templates simple and avoid complex logic
- Use native control flow (`@if`, `@for`, `@switch`) instead of `*ngIf`, `*ngFor`, `*ngSwitch`
- Use the async pipe to handle observables
- Do not assume globals like (`new Date()`) are available.

## Services
- Design services around a single responsibility
- Use the `providedIn: 'root'` option for singleton services
- Use the `inject()` function instead of constructor injection
